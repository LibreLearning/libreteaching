\section{Aspectos prácticos}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}

\begin{center}
\huge{Aspectos prácticos de las licencias libres}
\end{center}

\end{frame}



%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}
\frametitle{Escoger una licencia libre}


\begin{itemize}
\item Cada proyecto tiene sus objetivos y criterios en relación a la licencia.
\item El principal criterio de diferenciación es la existencia o no de pactos de de reciprocidad.
\item Otras licencias tienen un efecto copyleft reducido, que se aplica únicamente a la obra (o componente) original y a cualquier modificación de la misma. No se extiende a aplicaciones que ``integran''  o ``usan'' el componente libre. 
\item Políticas de doble licencia.
\end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}
\frametitle{Escoger una licencia libre}

¿Cuándo hacerlo?  

\begin{itemize}
\item Cuando queremos garantizar ciertas libertades básicas comunes a todo software libre.
\item Cuando queremos que una obra alcance el mayor uso y difusión posible.
\item Cuando queremos imponer algunas condiciones o restricciones (el reconocimiento de autoría, la falta de garantía, el uso de marcas, etc.
\item Cuando queremos mantener el control sobre la evolución del programa (copyleft).
\end{itemize}

\end{frame}


%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}
\frametitle{Escoger una licencia libre}


\begin{itemize}
\item Cada proyecto tiene sus objetivos y criterios en cuanto a licencia se refiere
\item El principal criterio de diferenciación es la existencia o no de pactos de de reciprocidad (copyleft).
\item Otras licencias tienen un efecto copyleft reducido, que se aplica únicamente a
la obra original y a cualquier modificación de la misma. No se extiende a aplicaciones que "integran" o "usan" el componente libre (copyleft débil). 
\end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}
\frametitle{¿Qué licencia escogería?}

Casos:

\small

\begin{itemize}
\item ¿Quiero permitir la privatización de obras derivadas y modificaciones?
\item ¿Quiero que los desarrolladores devuelvan sus modificaciones a la comunidad libre en
general, o a mí, como autor inicial, en particular?
\item ¿Quiero permitir que los licenciatarios fusionen o enlacen su programa con el mío?
\item ¿Quiero una mayor difusión del programa y tratar de establecer un estándar?
\item ¿Mi programa debe ejecutarse con otro en particular? ¿Tiene éste restricciones?
\item ¿Hay riesgo de que alguien tenga o pida una patente sobre un elemento
del programa?

\end{itemize}


\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}
\frametitle{Aplicar una licencia libre}


\begin{itemize}
\item Al inicio de cada archivo fuente 
\item Debería tener al menos la línea de ``copyright'' y un enlace a la versión completa del aviso.
\item Incluye además información de cómo contactar contigo por correo electrónico y ordinario.
\item Si el programa es interactivo, que muestre un breve aviso de copyright cuando se inicie en modo interactivo.
\item Si tiene ventanas o GUI, en la pestaña ayuda se puede añadir dicha nota de copyright (o incluso la licencia completa).
\end{itemize}

\end{frame}

\begin{frame}
\frametitle{Compatibilidad entre licencias}


\begin{itemize}
\item Dos licencias son incompatibles si no es posible combinar código de ambas obras respetando los términos de ambas licencias al mismo tiempo.
\item Afecta a la distribución, no al uso.
\item Que dos licencias sean libres no implica que sean compatibles.
\item Las licencias con copyleft son incompatibles entre ellas, salvo pacto explícito.
\item Compatibilidad ``por enlace'': aun si no se permite mezclar o integrar software con diferentes licencias, quizá sí se puedan enlazar.
\end{itemize}


\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}
\frametitle{Bifurcación (forks)}


\begin{itemize}
\item La bifurcación de un programa ocurre cuando el software se modifica y esta modificación se
desarrolla de manera separada, con otro equipo de desarrollo, y se distribuye con otro nombre, y quizá otra licencia.
\item Los forks solo son posibles con el software libre.
\item El software con GPL tiende a no dividirse: hay que mantener la licencia original.
\item El software con licencia tipo BSD se dividirá con mucha facilidad.
\item Los forks son casos extremos, y en general no son deseables (dividen esfuerzos, redundancia...).
\end{itemize}


\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}
\frametitle{Licencias libres y patentes}


\begin{itemize}
\item Patentes de programación: legales solo en EE.UU y Japón.
\item Protegen métodos o algoritmos. No permiten la reingeniería (el \textit{clean-room development})
\item No ofrecen protección extra y son un peligro para el software libre.
\item Para minimizar el riesgo de las patentes, las licencias libres más modernas disponen de cláusulas para otorgar licencias cruzadas de patentes entre contribuyentes y desarrolladores y pactos resolutorios para el caso en que se iniciara un litigio basado en una patente contra otro contribuyente. 

\end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}
\frametitle{Licencias y marcas}


\begin{itemize}
\small
\item El derecho de marcas protege las marcas registradas, que indican la conexión entre una persona o empresa y sus bienes y servicios, y los diferencian de los productos de otra persona.
\item Es importante notar que la marca registrada es un derecho separado del software (y de las licencias). 
\item El registro de una marca sobre un software libre sirve para respaldar la autoría original (reputación) y gestionar la evolución del software.
\item Algunas licencias obligan a los usuarios a indicar la autoría y la marca (Apache), otras prohíben su uso sin el consentimiento expreso del titular de la marca (Zope), otras impiden su uso sobre obras derivadas... 
\item El modelo de distribución y comercialización profesional del software libre depende en gran medida de las marcas (Linux\texttrademark, Red Hat\texttrademark, SuSE\texttrademark, OpenSolaris\texttrademark, Zope\texttrademark, Apache\texttrademark, Java\texttrademark, MySQL\texttrademark, etc.)

\end{itemize}


\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\begin{frame}
\frametitle{Licencias y garantía}


\begin{itemize}
\item Son habituales las cláusulas de exclusión de garantías y limitaciones de responsabilidad.
\item Hay dudas legales sobre la efectividad de estas cláusulas, que no serían válidas ante los consumidores.
\item Sí lo serían cuando no hay servicio prestado.
\item las licencias privativas usan cláusulas análogas, es un mito que acepten mayor nivel de responsabilidad.
\item Hay que considerar las garantías obligatorias por ley, que se aplican tanto al software libre como al software
privativo.
\item Las licencias libres permiten a los distribuidores de software libre agregar cláusulas de garantía.

\end{itemize}


\end{frame}
